## 发现的 Bug 与建议修复

### Bug 1：第 3 次草稿被 best_draft 吃掉（agent.py:312）

**症状：** `_reflection_count` 达 `MAX_REFLECTIONS` 时，下一稿被解析验证后，被 `final_result = best_draft if best_draft is not None else result` 丢弃，返回的是更早评分最高的稿。第 3 稿听了两轮批评专门改过的版本——没机会被采纳。

**触发证明：** `test_best_draft.py::test_third_draft_swallowed_by_best_draft`（PROVE——正好记录当前的 buggy 行为）

**建议修复（二选一）：**
- A) 给最后一次草稿一次反思机会（哪怕只用于打分）：`if _reflection_count < settings.MAX_REFLECTIONS or best_draft is None`
- B) 在比较条件里实际比对分数：`final_result = result if score_of_current >= best_score else best_draft`

### Bug 2：asyncio.gather 默认异常传播（agent.py:232）

**症状：** 5 个 tool 并行，1 个抛错，整个请求废。返回给用户的是异常而不是"其他 4 个工具的降级结果"。

**触发证明：** `test_asyncio_gather.py::test_gather_propagates_first_exception`（PROVE）

**建议修复：** `await asyncio.gather(..., return_exceptions=True)`，然后逐个判断 result 是否是 Exception 实例，失败工具返回错误字符串而非中断全请求。

### Bug 3：限流 IP key 内存泄漏（dependencies.py:11-20）

**症状：** 窗口内的请求数被正确限制，但每个 IP 的 key 永不被回收。即便所有时间戳过期，dict 项留着占内存。1000 IPs × 空list 仍然存了 1000 个 key。

**触发证明：** `test_rate_limit.py::test_ip_key_never_evicted`（PROVE）

**建议修复：** 周期性 sweep 删空列表的 key；或生产环境用 Redis + `EXPIRE`。

### Bug 4：限流多 Pod 失效（dependencies.py 设计层面）

**症状：** 进程内 dict 在多实例部署下分片。攻击者每加一个 Pod 就多获得一份配额的可用额度（限额 × pod 数）。

**触发证明：** `test_rate_limit.py::test_multi_instance_simulated`（PROVE）

**建议修复：** 外部共享计数器（Redis INCR + EXPIRE），或在网关层做限流。

### Bug 5：user_id 未鉴权（memory_service.py + models/request.py:20）

**症状：** 客户端伪造任意 user_id 即可读到所有对应用户的历史。零认证的越权读取。

**触发证明：** `test_memory_security.py::test_arbitrary_user_id_reads_anyone_history`（PROVE）

**简历表述修正：** "persistent per-user memory" → "session-scoped memory keyed by client-provided identifier"  
**建议修复：** 加服务端签名 cookie 或 HMAC(user_id)。

### Bug 6：LRU cache 不语义匹配（rag_service.py:40）

**症状：** `"敏感肌补水"` 和 `"敏感皮补水"` 两次都 miss —— lru_cache 使用 `(args, kwargs)` 的 hash 做 key，不是向量距离。

**触发证明：** `test_rag_stability.py::test_lru_does_not_match_paraphrase`（PROVE）

**建议修复：** 天然语言用户输入命中率本身接近 0，最简单的修复是**删掉 RAG 层 LRU**，仅保留 agent.py 内的 per-request `_tool_cache`（两层 cache 本来就是冗余的）。

## A/B 实验结果

完整报告：`docs/testing/2026-05-18-ab-report.md`  
原始数据：`scripts/ab_results.jsonl`（60 条 JSONL）

### 汇总

| 指标 | A (planning ON) | B (planning OFF) |
|---|---|---|
| 成功率 | **24/30 (80%)** | 15/30 (50%) |
| Reflection 最低分 | **7.0** ± 0.78 | 6.64 ± 0.84 |
| 平均总 tokens | **5,753** | 9,678 |
| 平均迭代数 | **2.38** | 3.73 |
| 平均墙钟时间 | **15.2s** | 16.6s |

### 配对比较（n=14 完整对）

| 指标 | 配对差均值 (A-B) | A 胜 | 平 | B 胜 |
|---|---|---|---|---|
| Reflection 最低分 | +0.39 | 5 | 6 | 2 |
| 总 tokens | -6,012 | 4 | 0 | 10 |
| 迭代数 | -1.71 | 3 | 1 | 10 |
| 墙钟秒数 | -3.36 | 4 | 0 | 10 |

### 结论

**Planning 阶段显著提升了 agent 的稳定性和效率，且多消耗的成本被下游节省完全抵消。**

- 成功率从 50% 提至 80%（+60% 相对提升）——这是最大的收益项
- 总 token 消耗反降 40%（5753 vs 9678）——一次 planning call 的成本完全被"少兜圈子"赚回来
- 平均迭代数减半（2.38 vs 3.73）
- Reflection 评分略高（+0.36），但偏差范围小、且受 self-eval bias 影响，标记为"温和利好"而非主证

**待验证：** 21 个失败案例（主要出现在某些 query 类别上，两臂都有），需要独立 judge 模型 + 更大样本才能确认 B 路的质量劣势是否真实。目前只能说：planning 对可靠性的改善是实质性的。

## 附加实验：DuckDuckGo → Tavily 工具迁移

在 A/B 实验进行之前执行了一次紧急工具修复。诊断显示原 DDGS 搜索层是 agent 失败的主因（搜索返回 ASMR 无关内容 + 网页抓取空响应）。更换为 Tavily 后，同样一条以前 100% 失败的 query 在新的搜索后端上第一次通过了。

对比数据（qid=1, no planning, max_iterations=5）：

| 场景 | 工具表现 | 结果 |
|---|---|---|
| DDGS | get_trending "空"、search 返回 ASMR、fetch "页面内容为空" | 5 轮全在调工具，最终失败 |
| Tavily | get_trending "发现报告、种草"、search 返回珀莱雅真实测评、fetch 返回完整内容 | 4 轮调工具后输出合法 JSON，成功 |

## 复盘

1. **best_draft 的 fallback 比较条件代码与注释漂移。** commit 当时的注释写 "fall back to current"，但代码实际上选了 best_draft 而不计 current。这在新人接手时一定会踩坑。
2. **A/B 实验是验证设计假设的最便宜手段。** 30 对查询、约 $2 的 DeepSeek 费用 + Tavily 免费额度，就让"planning 能不能用"这个问题从感觉变成了数据——而且结果还跟直觉反了过来（planning 居然是省 token 的）。
3. **PROVE 类测试是面试财富。** 把已知 bug 编成"故意通过"的测试用例，面试里能从代码出发讲清楚 bug 的行为、影响、修复路径，比"这个项目有哪些不足"的口头回答强 10 倍。
4. **搜索后端的质量直接决定 agent 的可用性。** DDGS 换成 Tavily 后同一条 query 从 100% 失败变 100% 成功。Agent 架构再好，工具不可靠它就一样不行。这跟"垃圾进垃圾出"是一个道理。
